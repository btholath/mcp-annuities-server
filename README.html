Step-by-step — what each layer does

1
User terminal Domain 3
VS Code integrated terminal running bash inside WSL/Ubuntu

You type a natural-language request to Claude Code — for example "Use the annuities-server to look up CLIENT_0001". Several things are already in place before this text goes anywhere:

Shell context. Your terminal is running inside ~/mcp-annuities-server. Claude Code calls the absolute path to .venv/bin/python directly, which is why fixing the path from /usr/bin/python3.12 (system Python, no mcp package) to /home/bijut/mcp-annuities-server/.venv/bin/python (venv, packages installed) was the critical fix.

Auth context. The session is authenticated via Claude Pro subscription (not the API key) after running claude /logout and re-logging in without approving the API key. This avoids the "Both claude.ai and ANTHROPIC_API_KEY set" warning and keeps Claude Code usage off your API billing.

What gets sent upstream. Your text, conversation history, CLAUDE.md memory files, and the full list of tool schemas from all connected MCP servers are bundled into the model call.

The terminal is just the I/O surface. All orchestration (deciding which tool to call, building the JSON-RPC request, managing the subprocess) happens inside the claude process, not in bash.
2
Claude Code reasoning Domain 1 Domain 2
Tool schema matching · model: Sonnet 4.6 / Opus 4.8

This is the orchestration layer where a plain-English sentence becomes a structured decision about which tool to call and with what arguments.

Step 2a — Assemble context. Claude Code builds the full message payload: your request, conversation history, CLAUDE.md instructions, and the tool schemas from annuities-server (5 tools with JSON Schema definitions — parameter names, types, descriptions).

Step 2b — Tool selection. The model reads "look up CLIENT_0001" against the tool descriptions and matches it to get_client(client_id: string). This is pattern matching against schema text — which is exactly why good descriptions matter.

Step 2c — Produce tool_use block. Instead of a prose response, the model emits:

{"type":"tool_use","name":"get_client","input":{"client_id":"CLIENT_0001"}}

Step 2d — Wait for result. Claude Code pauses the model's turn, sends the call down to layer 3, and feeds the result back as a tool_result block so the model can write its final formatted answer.

CCAF exam note (Domain 2): Tool descriptions are the only signal the model uses for selection. A vague description like "get data" causes wrong tool picks. A specific one like "Retrieve full annuity contract details for one client by ID" is reliable. This is the "tool boundary design" concept tested in Domain 2.
3
MCP stdio handshake Domain 2
JSON-RPC 2.0 over child process stdin / stdout pipes

This is where Claude Code's abstract "call get_client" decision becomes an OS-level connection to your server.py process.

Subprocess spawn (once, at session start). When Claude Code loaded ~/.claude.json and saw the annuities-server entry, it immediately spawned /home/bijut/mcp-annuities-server/.venv/bin/python server.py as a child process with three wired pipes: stdin, stdout, stderr. The process stays alive for the whole session — this is why the manual sanity check (python server.py hangs silently) works.

Initialization handshake. Before any tool can be called, the MCP client sends an initialize request over stdin. FastMCP responds with its capabilities (the tool/resource/prompt list). This is what claude mcp list checks when it reports ✔ Connected.

Message framing. Every message is a single line of newline-delimited JSON-RPC 2.0:

{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"get_client","arguments":{"client_id":"CLIENT_0001"}}}

stdout is reserved exclusively for protocol messages — print() in server.py would corrupt the stream. Logging goes to stderr (which is why INFO lines appeared in your test output without breaking anything).

Lifetime. The subprocess stays alive for the whole Claude Code session. The earlier ✘ Failed to connect happened because realpath resolved the venv symlink to /usr/bin/python3.12, which failed import mcp immediately — so the handshake in this step never completed.

CCAF exam contrast (Domain 2 — SSE transport): With SSE transport, instead of pipes to a local subprocess, the client opens an HTTP connection to a server anywhere on the network. The JSON-RPC message content stays identical; only the transport changes. SSE is multi-client; stdio is single-client.
4
FastMCP dispatch + validation Domain 2
server.py receives the JSON-RPC message and routes to the right function

1. Event loop reads the line. mcp.run() (last line of server.py) continuously reads newline-delimited JSON from stdin.

2. Method dispatch. FastMCP checks method: "tools/call", looks up name: "get_client" in its internal registry — built at startup by scanning every @mcp.tool() decorated function.

3. Schema validation. FastMCP validates arguments against the JSON Schema auto-generated from the function's type hints before calling anything. Wrong type or missing required field → error response, no Python code runs.

4. Function call. get_client(client_id="CLIENT_0001") — plain Python, synchronous.

5. Result serialization. The returned dict is wrapped into an MCP CallToolResult with a text content block (even dict returns become JSON-encoded text — MCP content types are text/image/resource).

6. Write to stdout → layer 3 reads it → layer 2 gets the result → model writes its final answer → layer 1 shows it to you.

Why FastMCP over the low-level Server class: With @mcp.tool(), schema generation and validation are automatic from type hints. The original mcp-prompt-templates-main used the low-level class — you'd hand-write the JSON Schema in list_tools() and manually validate arguments in call_tool(). More boilerplate, more chance of schema/function drift.
5
Tool execution Domain 2 Domain 4
get_client and calculate_payout — what each function actually does

get_client(client_id: str) → dict

@mcp.tool()
def get_client(client_id: str) -> dict:
    for row in _load_all():
        if row["client_id"] == client_id:
            return row
    return {"error": f"Client {client_id} not found"}

Calls _load_all() which opens data/annuities.csv via csv.DictReader — every value comes back as a string, even numeric fields like age or premium_amount. Linear-scans for a matching client_id. Re-reads the entire CSV from disk on every call (fine at 500 rows; a production version would cache or index).

calculate_payout(principal, annual_rate_pct, term_years) → dict

monthly_rate = (annual_rate_pct / 100) / 12
monthly_payment = principal * monthly_rate / (1 - (1 + monthly_rate) ** (-n_payments))

Standard amortization formula (same math as a mortgage payment). No data lookup — pure calculation. Important design choices: input validation first (term_years <= 0 returns a structured error), and the zero-rate edge case is handled separately to avoid dividing by zero.

The chaining insight. When you said "calculate the payout for this client", Claude Code had no single tool that does both. It implicitly chained: get_client → extract current_account_value + crediting_rate_pct → calculate_payout. This multi-step reasoning from one natural-language sentence is the minimal live version of Domain 1's agentic loop — no orchestrator script needed.

Subtle quirk: two different "monthly payment" numbers exist. monthly_payment in the CSV is the original amortized payment computed at contract issue. calculate_payout computes a new payout from current_account_value today. These differ. Claude correctly flagged this in your session: "the payment frequency is listed as Quarterly while there's also a monthly payment field — let me know if you'd like me to calculate the payout."
Domain 4 connection: Every value is stored as a string — CSV has no types. This is why Pydantic structured output schemas are valuable: a model with premium_amount: float coerces "77882.26" automatically. Without a schema, every agent or tool doing math must manually call float(row["premium_amount"]).
