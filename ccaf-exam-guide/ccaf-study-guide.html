Domain 1 — Agentic Architecture & Orchestration
~16 of 60 questions · Most underestimated domain · Judgment over memorisation

The agentic loop — exact lifecycle

Critical: Always terminate on stop_reason === "end_turn". Never parse text content to detect completion. This is the single most-tested anti-pattern in Domain 1.
# The canonical 6-step loop
while True:
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=4096,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "end_turn":    # ← PRIMARY STOP
        return response.content[0].text

    if response.stop_reason != "tool_use":
        raise RuntimeError(f"Unexpected: {response.stop_reason}")

    # Step 4: append assistant message FIRST
    messages.append({"role": "assistant", "content": response.content})

    # Step 5: execute ALL tool calls
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": json.dumps(result)
            })

    # Step 6: tool results go back as USER message
    messages.append({"role": "user", "content": tool_results})

Orchestration patterns

🏛️ Hub-and-spoke (most tested)
One coordinator → N specialist subagents. All inter-agent communication routes through the hub. Subagents have fully isolated context — they see only what coordinator explicitly passes. Parallel spawning = multiple Task calls in ONE response turn.
⛓️ Pipeline (sequential)
Agent A output → Agent B input → Agent C. Use ONLY when each step genuinely depends on the prior output. Choosing pipeline when steps are independent wastes latency — the exam tests this tradeoff.
⚡ Parallel fan-out
Independent subtasks run simultaneously. Emit ALL Task calls in one coordinator response turn = concurrent. Separate turns = sequential. This distinction is a frequent exam question.
🔄 Evaluator-optimizer
Generator → Evaluator judges quality → Coordinator re-delegates if criteria not met → repeat until quality bar or MAX_ROUNDS. Always set a ceiling to prevent infinite loops.

Hooks & enforcement

Hooks = 100% compliance. Prompt rules = ~99%. For financial gates, PII checks, and security policies, hooks are always the architecturally correct answer.

Domain 2 — Tool Design & MCP Integration
~11 of 60 questions · Build a server — don't just read about one

MCP server anatomy

🔧 Tools
Functions Claude can call. Defined with JSON Schema input_schema. Named, typed, validated before execution. Keep count to ≤7 — too many tools degrades tool-selection reliability.
📄 Resources
Static or dynamic data the server can serve. URI-addressed (e.g. annuities://dataset). Client reads them directly — no tool call overhead. Good for large reference data.
📝 Prompts
Reusable prompt templates the server exposes. YAML-driven (config.yaml + template.md). Loaded on-demand — zero context cost until invoked. Good for domain-specific workflows.
# FastMCP — the idiomatic server pattern
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("annuities-server")

@mcp.tool()
def get_client(client_id: str) -> dict:
    """Retrieve full annuity contract details for one client by ID."""
    # Description text = what Claude uses for tool selection
    ...

@mcp.resource("annuities://dataset")
def get_dataset() -> str:
    return DATA_FILE.read_text()

@mcp.prompt("annuity_review")
def annuity_review(client_id: str) -> str:
    _, template = load_template("annuity_review")
    return fill(template, client_id=client_id)

Tool schema design rules

Too many tools breaks tool selection. The exam will test you on this. Keep tool count ≤7. Every tool added after ~5 increases the probability of Claude picking the wrong one. This is the "tool boundary / reasoning overload" concept.
RuleReason
Description must be specific and action-orientedTool selection is pattern-matching against description text
Use enum for fields with fixed valid valuesPrevents invalid values without runtime validation code
Mark genuinely optional fields with required: [] exclusionForces Claude to pass required args — API rejects missing ones
Return structured errors, not exceptionsUnhandled exceptions produce opaque MCP errors Claude can't reason about
Never print() in server.pystdout is reserved for JSON-RPC protocol — print corrupts the stream

Domain 3 — Claude Code Configuration & Workflows
~12 of 60 questions · Hands-on configuration, not theory

CLAUDE.md hierarchy

CLAUDE.md files are loaded in order from most global to most specific. All non-conflicting rules from all levels apply simultaneously. More specific files win on conflicts.

# Precedence order (most specific wins on conflicts)
~/.claude/CLAUDE.md              # 1. User-global — applies everywhere
~/project/CLAUDE.md              # 2. Project root — applies to whole project
~/project/domain1/CLAUDE.md      # 3. Subfolder — applies to that domain
~/project/domain1/sub/CLAUDE.md  # 4. Deep subfolder — most specific
Token cost: Every CLAUDE.md level costs tokens on every turn. The /context command shows "Memory files: N tokens". Keep each CLAUDE.md under 500 tokens — verbose files dilute Claude's adherence to the rules.

Domain 4 — Prompt Engineering & Structured Output
~12 of 60 questions · Schema enforcement, retry loops, batch API

JSON schema enforcement

Format instructions belong in the SYSTEM prompt, not the user prompt. This is the most-tested Domain 4 rule. System prompt = format and schema. User prompt = the data to process.
# System prompt carries the schema
SYSTEM_PROMPT = f"""You are an annuity recommendation engine.
Return ONLY a JSON object matching this exact schema:
{json.dumps(AnnuityRecommendation.model_json_schema(), indent=2)}

Rules:
- recommended_action: HOLD | EXCHANGE | SURRENDER | CONVERT
- urgency: HIGH if surrender_charge_pct > 7 or age > 70
- confidence_score: 0.0–1.0
- No other text, no markdown fences."""

Validation retry loop

def get_recommendation(record: dict, max_retries=3):
    messages = [{"role": "user", "content": json.dumps(record)}]
    last_error = ""

    for attempt in range(1, max_retries + 1):
        if last_error:
            # Feed SPECIFIC error back — not just "try again"
            messages.append({"role": "assistant", "content": last_error})
            messages.append({"role": "user", "content":
                f"Validation failed: {last_error}\n"
                "Fix the specific field and return valid JSON."})

        resp = client.messages.create(
            model=MODEL, max_tokens=512, system=SYSTEM_PROMPT, messages=messages)
        raw = resp.content[0].text.strip()

        try:
            data = json.loads(raw)
            result = AnnuityRecommendation(**data)   # Pydantic validates
            return result                           # ✅ success
        except (json.JSONDecodeError, ValidationError) as e:
            last_error = str(e)[:200]               # capture for next turn

    raise RuntimeError(f"Failed after {max_retries} attempts")

Domain 5 — Context Management & Reliability
~9 of 60 questions · Context efficiency, caching, confidence calibration
Don't skip Domain 5. Context, caching, and confidence concepts recur inside Domain 1 and Domain 4 scenarios too. Candidates who skip this domain because "it's only 15%" lose points across the whole exam.

Handoff patterns between agents

StrategyWhat is passedWhen to use
Structured summary (HandoffPackage)Typed dataclass with only what recipient needsDefault — prevents context explosion
Sliding windowLast N turns, older turns droppedLong conversations, historical depth less important
Explicit extractionAsk Claude to extract key facts before handoffWhen prior conversation has relevant facts buried in prose
Never pass full chat history to a subagent. It causes context explosion and includes irrelevant noise. The HandoffPackage pattern — a lean, typed, structured object — is the correct answer on every handoff scenario question.
@dataclass
class HandoffPackage:
    client_id: str
    age: int
    product_type: str
    key_findings: list[str]    # max 5 bullet points
    flags: list[str]           # issues requiring attention
    confidence: float          # 0.0–1.0
    escalate_to_human: bool    # if True, bypass AI

Prompt caching

ConceptDetail
Cache breakpointsUse cache_control: {type: "ephemeral"} to mark cacheable content
What to cacheLarge static context: system prompts, reference docs, tool definitions
What NOT to cacheRapidly changing content, per-user data, short content (cache overhead > savings)
Cost modelCache write: 25% more tokens. Cache read: 90% cheaper. Break-even ~4 reads.
TTLEphemeral: 5 min. Persistent: varies by plan. Re-use within TTL = cache hit.

Confidence calibration

# Confidence threshold pattern
CONFIDENCE_THRESHOLD = 0.65

class CalibratedOutput(BaseModel):
    answer: str
    confidence: float          # 0.0 = certain wrong, 1.0 = certain correct
    reasoning_basis: str       # "calculated" | "estimated" | "inferred" | "unknown"
    data_gaps: list[str]       # what information was missing
    escalate: bool             # auto-set: confidence < CONFIDENCE_THRESHOLD

# Agents must NEVER silently guess on high-stakes outputs
if output.confidence < CONFIDENCE_THRESHOLD:
    route_to_human_reviewer(output)
